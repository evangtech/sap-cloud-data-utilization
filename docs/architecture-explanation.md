## 設計原則：構造データとイベントデータの分離

| | Neptune Analytics（KG） | DynamoDB |
|---|---|---|
| **格納対象** | サプライチェーン構造 | リスクイベント＋算出済み影響 |
| **データ性質** | 比較的静的（ERP更新時に変更） | 時系列（発生→解決のライフサイクル） |
| **クエリパターン** | グラフ探索（多ホップの関係性追跡） | 時間帯・ステータスによるフィルタ |
| **例** | 「Material Xを供給するサプライヤーは？」 | 「現在アクティブなリスクイベントは？」 |
|  | 「Product Yの部品構成（BOM）は？」 | 「過去7日間のイベント一覧」 |
|  | 「Supplier Zの代替候補は？」 | 「影響総額の推移」 |

### なぜRiskEventをNeptuneに入れないのか

ご提案いただいたKG設計ではRiskEventをノード、IMPACTS_*をエッジとして格納する構成ですが、代替案を提案します。

**IMPACTS_*エッジは計算結果であり、構造的知識ではありません。** 関税イベント発生時、まずNeptuneを照会して影響サプライヤーを特定し、その結果をNeptuneにIMPACTS_*エッジとして書き戻し、ダッシュボードで再度読み出すことになります。これは既に算出済みのデータを往復させる処理です。

代わりに：
1. イベント到着 → AIパイプラインがNeptune構造グラフを照会 → 影響を算出
2. イベント＋算出結果をDynamoDBにスナップショットとして保存
3. フロントエンドはDynamoDB（イベント・影響）とNeptune（構造データ）をそれぞれ参照

**メリット：**
- イベント毎のグラフ変更が不要（Neptuneはクリーンな状態を維持）
- イベント解決時にIMPACTS_*エッジの削除処理が不要
- DynamoDBの時間ベースクエリがイベントライフサイクル管理に適合
- 現在の運用状態は常にアクティブイベント群から導出可能

## イベント積層：複数イベントの相互作用

新しいリスクイベント到着時、既存のアクティブイベント全体を考慮して影響を算出します。

**例：**
```
Day 1: Supplier A近辺で地震 → AI推定: 生産能力40%低下
Day 3: 中国半導体に関税 → Supplier B（中国）が影響
        → Supplier AはSupplier Bの代替候補だった
        → しかしSupplier Aは既に生産能力60%（Day 1の地震）
        → 総合判定: クリーンな代替先なし → Criticalアラート
```

**実装：**
1. 新イベント到着
2. パイプラインがDynamoDBから全アクティブイベントを読込
3. 構造グラフに対して新イベントの影響を算出
4. 既存アクティブイベントの影響と突合
5. 統合影響スナップショットを生成
6. イベント解決時 → 残存アクティブイベントから再算出

**現在状態 = f(構造グラフ × アクティブイベント群)**

## AIの役割：3つの利用箇所

### 1. イベント解析（入力側）
非構造化テキストを構造化パラメータに変換。
```
入力: 「米国が中国半導体に15%関税を賦課」
出力: {type: TARIFF_CHANGE, country: CN, hs_chapter: 85, rate_change: +0.15}
```

### 2. 不確実な影響の推定
確定値がないイベントの影響度をAIが推定。
```
入力: 震度6.5、工場から50km、インフラ種別: 工業
出力: 推定生産能力低下: 35-45%、確信度: 0.7
```

### 3. 要約生成（出力側）
算出結果から自然言語のサマリーを生成。
```
入力: {影響サプライヤー3社, 品目5件, 影響総額: ¥4.2億, 代替不可: 2製品}
出力: 「この関税変更により3社のサプライヤーが影響を受け...」
```

**ステップ2〜4（Neptune照会・影響額算出・代替候補探索）はルールベースであり、AIは不要。**

## 画面別のデータフロー

### 画面A：リスクダッシュボード
```
DynamoDB → アクティブイベント一覧、アラート件数、影響金額
Neptune  → サプライヤー・工場の位置情報（地図プロット用）
```

### 画面B：影響分析・代替提案
```
DynamoDB → 選択イベントの影響スナップショット（影響サプライヤー・品目リスト）
Neptune  → 構造的探索で代替候補を取得＋スコアリング
```

### 画面C：What-ifシミュレーション
```
Neptune → BOM構成、現行関税率、サプライヤー単価
Lambda  → OR-Tools / 計算エンジンで利益シミュレーション
```

## GraphRAGの位置づけ

GraphRAGは**構築時**ではなく**クエリ時**のパターンとして活用します。全画面共通の自然言語クエリバーに適用：

```
ユーザー: 「この関税変更がQ2売上にどう影響する？」
  ↓
nl-query Lambda:
  1. Bedrockが質問を解析 → openCypherクエリ生成
  2. Neptuneが関連サブグラフを返却（影響製品、顧客、受注金額）
  3. Bedrockがサブグラフをコンテキストとして自然言語回答を生成
```

本PoCのKGはSAP・WTOの構造化データから構築するため、文書からのグラフ自動構築（Microsoft GraphRAG）は不要です。

## 確認事項

1. **イベントライフサイクル**: 解決済みイベントの保持期間は？（DynamoDB TTL設定に影響）
2. **AIエージェント vs パイプライン**: イベント処理をStep Functions（確定的パイプライン）とするか、Bedrockエージェント（ツール付き自律型）とするか。PoCではどちらが望ましいでしょうか。
3. **画面Cのデプロイ**: Streamlit指定ですが、別Amplifyアプリとしてサイドバーナビゲーションからアクセスする構成でよろしいでしょうか。
4. **影響推定の確信度**: AI推定値（例: 地震の影響度）の確信度スコアをユーザーに表示すべきでしょうか。
